**提案编号：** P-20260417-001

**生成日期：** 2026-04-17
**状态：** prd_pending_confirmation

## 1. Overview

### 背景

当前 AI 编程工具市场已被 Claude Code、Cursor、WindSurf 等产品定义。开发者对"真正可用的 AI 编程助手"的需求持续增长——不是补全代码的半成品，而是能理解意图、规划路径、执行任务、验证结果的整体系统。

Boss 提供了 Harness Engineering 的详细资料，包含六大核心组件的系统设计。本 PRD 旨在将 Harness Engineering 的核心理念落地为一款面向 Windows 平台的 AI 编程桌面应用。

### 问题

- 现有 AI 编程工具在复杂、长链路任务上表现不稳定，缺乏统一的规划-执行-验证闭环。
- 大多数工具将上下文管理、工具调用、安全验证耦合在提示词层面，缺乏工程化的分层架构。
- Windows 平台缺乏一款真正基于 Harness Engineering 理念构建的原生桌面产品。

### 机会

基于 Harness Engineering 的六大组件，构建一款架构清晰、可扩展的 AI 编程工具，在规划能力、上下文管理、工具调用可靠性上形成差异化竞争力。

---

## 2. Goals

### 业务目标

- 打造一款可在 Windows 平台稳定运行的 AI 编程桌面应用，核心技术能力对标 Claude Code。
- MVP 版本实现完整的 Tool Call Loop（规划+执行闭环），验证 Harness Engineering 核心架构的可行性。
- 为后续接入多模型、支持 Sub Agent、扩展工具集打下可扩展的架构基础。

### 用户目标

- 开发者可通过自然语言指令，完成文件读写、代码修改、Bash 命令执行等日常开发任务。
- 长链路任务（如"重构整个模块"）系统先给出执行方案，用户确认后逐步执行。
- 工具调用结果由后台验证机制客观评估，减少 AI 幻觉导致的错误执行。

---

## 3. Non-Goals

- **不做**深度 IDE 集成（如 VS Code 插件形态），MVP 为独立 Electron 桌面应用。
- **不做**多模型路由或模型托管，MVP 使用外部 API Key 接入模型服务。
- **不做**Sub Agent 能力，留在 MVP 后迭代。
- **不做**完整的移动端或 Web 适配，专注 Windows 桌面体验。
- **不做**插件市场或开放工具注册体系，工具集在训练阶段固定。

## 4. Target Users

### 主要用户

| 用户画像 | 描述 |
|---|---|
| 全栈开发者 | 需要在 Windows 环境下频繁处理文件、运行脚本、执行构建命令 |
| DevOps/后端工程师 | 习惯 CLI 操作，需要自动化脚本和批量任务执行能力 |
| AI 编程工具早期采用者 | 对 Claude Code、Cursor 有使用经验，希望尝试新工具 |

### 典型用户场景

1. **日常代码辅助**：在项目中用自然语言询问代码逻辑、生成单元测试、解释陌生代码段。
2. **批量文件修改**：将某目录下所有文件的命名风格从 camelCase 改为 snake_case。
3. **长链路重构任务**：先生成重构方案（规划模式），用户确认后逐步执行并汇报进度。
4. **开发环境维护**：执行 `npm install`、`pytest`、Git 操作等日常命令。
5. **上下文感知修改**：基于项目当前文件内容进行修改，而非脱离上下文的泛化生成。

## 5. Scope

### MVP 六大组件实现优先级

| 组件 | MVP 是否包含 | 说明 |
|---|---|---|
| 多层级 System Prompt | ✅ 包含 | 固定缓存层 + 动态替换层，MVP 简化实现 |
| Tool Schema | ✅ 包含 | 内置核心工具集，支持并行调用；权限分级 |
| Tool Call Loop | ✅ 包含 | 规划模式 + 执行模式，MVP 重点实现 |
| Context Manager | ✅ 包含 | 指针索引 Memory，后台合并/去重/关联 |
| Sub Agent | ❌ 暂不包含 | v2 迭代计划 |
| Verification Hooks | ⚠️ 简化包含 | MVP 仅实现基础的执行结果校验，不含后台分类器 |

### MVP 功能范围

**必须实现（Must Have）**

1. 独立的 Electron 桌面应用窗口，基础 UI（聊天式对话界面 + 结果展示）
2. 多层级 System Prompt 引擎，支持动态注入会话状态、当前时间、打开文件列表
3. 核心工具集：文件读取、文件写入、目录列表、Bash 命令执行、搜索
4. Tool Schema 规范定义，含权限分级（低风险/中风险/高风险）
5. 规划模式：长任务先生成执行方案，用户确认后进入执行
6. 执行模式：沙盒环境中按规划执行工具调用，实时返回结果
7. 指针索引式 Memory：维护上下文指针，支持上下文压缩与扩展
8. 基础验证机制：工具执行后检查返回值格式与基本正确性

**可选实现（Should Have）**

1. 多标签页支持（每个项目独立上下文）
2. 快捷命令面板
3. 命令执行历史记录

**明确不做（Explicitly Out）**

1. IDE 插件形态
2. 多模型路由
3. Sub Agent
4. 插件市场

## 6. User Flow

### 主流程：单次任务执行

```
用户输入自然语言指令
        ↓
系统判断是否为长链路任务（>3步 or 包含多个文件）
        ↓
    ├── 否（短任务）
    │       ↓
    │   直接进入执行模式
    │       ↓
    │   调用 Tool Schema 中对应工具
    │       ↓
    │   Verification Hooks 校验结果
    │       ↓
    │   返回结果给用户
    │
    └── 是（长链路任务）
            ↓
        进入规划模式
            ↓
        生成执行方案（含步骤拆解、风险提示）
            ↓
        展示方案，等待用户确认/修改
            ↓
        用户确认
            ↓
        进入执行模式（按序/并行执行各步骤）
            ↓
        每步执行后触发 Verification Hooks
            ↓
        所有步骤完成后汇报最终结果
```

### 上下文管理流程

```
每次对话产生新的用户输入 + Tool 执行结果
        ↓
Context Manager 更新 Memory 指针
        ↓
后台任务：合并相似上下文、去重、关联相关文件
        ↓
当上下文超过阈值时，触发压缩（保留关键指针，压缩详细内容）
        ↓
后续对话基于压缩后的 Memory + 原始指针索引继续
```

## 7. Requirements

### 7.1 多层级 System Prompt

**需求描述：**

System Prompt 由固定缓存层和动态替换层组成。

- **固定缓存层**（每次会话不变）：身份定义、核心工具说明、安全策略、输出格式规范。
- **动态替换层**（每次请求时注入）：当前会话状态、当前时间戳、项目可读取文件列表（基于工作目录）、用户偏好设置。

**技术要求：**

- 固定缓存层在应用启动时一次性加载，后续请求复用。
- 动态层在每次 LLM 请求前由 Context Manager 注入。
- 支持通过配置文件自定义固定缓存层内容。

### 7.2 Tool Schema

**需求描述：**

内置以下核心工具，在训练阶段即完成适配，工具调用经过验证：

| 工具名称 | 功能 | 风险等级 |
|---|---|---|
| `file_read` | 读取指定路径文件内容 | 低 |
| `file_write` | 写入或覆盖文件 | 中 |
| `file_append` | 追加内容到文件尾部 | 中 |
| `dir_list` | 列出目录文件结构 | 低 |
| `bash_execute` | 在沙盒中执行 Bash 命令 | 高 |
| `grep_search` | 在文件中搜索关键字 | 低 |

**权限分级：**

- **低风险**：只读操作，默认允许。
- **中风险**：写入操作，执行前需要用户显式确认（可配置为免确认）。
- **高风险**：系统命令执行，默认阻止，首次使用需用户授权。

**并行调用：**

- Tool Schema 支持并行定义，允许多个工具同时调用（类似 Claude Code 的 `parallel_tool_calls`）。
- 并行执行结果合并后统一返回给 LLM。

### 7.3 Tool Call Loop

**需求描述：**

Tool Call Loop 包含两种工作模式：

#### 规划模式（Planning Mode）

触发条件：用户任务预计超过 3 个工具调用步骤，或涉及多个文件的修改。

行为：

1. LLM 接收任务后，首先生成**执行方案**（而非直接执行）。
2. 执行方案包含：步骤列表、每个步骤的工具调用、预期结果、潜在风险。
3. 方案以结构化格式展示给用户。
4. 用户可以修改方案中的步骤，或直接确认执行。
5. 用户确认后，切换到执行模式。

#### 执行模式（Execution Mode）

行为：

1. 根据规划（如果是规划模式）或直接分析（短任务）调用对应工具。
2. 工具在沙盒环境中执行，获取结构化结果。
3. 结果触发 Verification Hooks 进行校验。
4. 校验通过后，结果注入上下文，继续下一步（如果是多步骤）。
5. 所有步骤完成后，生成自然语言回复返回给用户。

**沙盒机制：**

- Bash 命令在受限的沙盒环境中执行。
- 沙盒限制：禁止网络访问（curl/wget 等），限制文件写入路径，禁止系统级命令（rm -rf 等）。
- 每个工具调用有超时机制（默认 30 秒，可配置）。

### 7.4 Context Manager

**需求描述：**

上下文管理器负责管理和维护对话上下文。

**核心机制：**

- **指针索引**：每次工具调用或用户输入后，Context Manager 为其分配唯一指针 ID。
- **Memory 结构**：存储 `<指针ID, 内容摘要, 详细内容>` 的映射，支持快速回溯。
- **自动合并**：检测相似或重复的上下文片段，自动合并并保留去重后的指针。
- **文件关联**：当工具调用涉及某文件时，自动关联该文件的所有历史操作记录。
- **上下文压缩**：当上下文 token 超过阈值（默认 80% 的模型上下文窗口），自动压缩低价值片段，保留关键指针。

**容量配置：**

- 支持通过配置文件设置上下文窗口上限（建议默认 128K tokens）。
- 支持手动触发上下文压缩。

### 7.5 Verification Hooks（简化实现）

**需求描述：**

MVP 阶段实现基础验证，不含后台分类器。

**验证内容：**

- **格式校验**：工具返回值是否符合 Tool Schema 定义的返回格式。
- **基本正确性校验**：例如 `file_write` 后验证文件确实被写入；`bash_execute` 返回码为 0。
- **边界校验**：文件路径是否在允许的工作目录范围内；命令是否包含禁止关键字。
- **结果差异检测**：如果工具返回结果与预期差异过大（如文件行数大幅变化），标记警告。

**验证结果处理：**

- 校验通过：正常流程继续。
- 校验失败/警告：记录到执行日志，并在回复中告知用户可能的问题。

## 8. Technical Architecture

### 8.1 技术栈

| 层级 | 技术选型 | 说明 |
|---|---|---|
| 桌面框架 | Electron | 跨平台桌面应用框架，Windows 优先 |
| 前端 UI | React + TypeScript | 主流前端框架，便于维护 |
| 状态管理 | Zustand | 轻量级状态管理 |
| 消息通信 | electron IPC | 主进程与渲染进程通信 |
| 沙盒执行 | Node.js child_process（受限） | 工具调用的安全执行环境 |
| LLM 接入 | 外部 API（OpenAI Compatible） | MVP 阶段通过 API Key 接入 |
| 配置文件 | JSON / YAML | 工具定义、权限配置等 |

### 8.2 系统架构图

```
┌──────────────────────────────────────────────────────┐
│                    Renderer Process                   │
│  ┌─────────────┐   ┌──────────────┐   ┌───────────┐  │
│  │   React UI  │ ←→ │ Context Mgr  │ ←→ │ LLM Bridge│  │
│  └─────────────┘   └──────────────┘   └───────────┘  │
└────────────────────────┬─────────────────────────────┘
                         │ IPC
┌────────────────────────┴─────────────────────────────┐
│                     Main Process                      │
│  ┌─────────────┐   ┌──────────────┐   ┌───────────┐  │
│  │Tool Executor│ ←→ │  Sys Prompt  │ ←→ │Sandbox Mgr│  │
│  └─────────────┘   │   Engine      │   └───────────┘  │
│                    └──────────────┘                   │
└──────────────────────────────────────────────────────┘
```

### 8.3 核心模块职责

| 模块 | 职责 |
|---|---|
| `LLMBridge` | 封装 LLM API 调用，注入 System Prompt，处理流式响应 |
| `SystemPromptEngine` | 管理固定缓存层与动态替换层，构建每次请求的完整 Prompt |
| `ToolExecutor` | 接收工具调用请求，在沙盒中执行，返回结构化结果 |
| `SandboxManager` | 管理沙盒环境生命周期，强制执行安全策略 |
| `ContextManager` | 维护 Memory 指针，处理上下文合并/压缩/关联 |
| `VerificationHooks` | 对工具执行结果进行格式和正确性校验 |
| `UIComponents` | 对话界面、规划方案展示、设置面板等 |

### 8.4 安全策略

- 所有文件操作限制在用户指定的工作目录内（配置文件可设置根目录白名单）。
- Bash 命令沙盒化执行，禁止高危系统命令（可配置禁止命令列表）。
- API Key 仅存储在本地，不经过任何第三方服务。
- 支持在设置中查看和撤销已授权的高风险工具调用权限。

## 9. Differentiation

### 与竞品对比

| 维度 | Claude Code | Cursor | WindSurf | **Harness Desktop** |
|---|---|---|---|---|
| 架构理念 | Tool Call Loop + 端到端 | IDE 深度集成 | 云端优先 | **Harness Engineering 六组件，系统化分层** |
| Tool Call Loop | ✅ | ⚠️ 较弱 | ⚠️ 较弱 | ✅ **规划+执行双模式，Verification Hooks** |
| Context Manager | ⚠️ 隐式 | ⚠️ 隐式 | ⚠️ 隐式 | ✅ **指针索引，主动合并/压缩/关联** |
| Sub Agent | ✅ | ❌ | ❌ | ❌ v2 支持 |
| 平台 | macOS/Linux/Windows | 跨平台 | Web/桌面 | **Windows 优先，Electron 原生** |
| 沙盒安全 | ✅ | ⚠️ | ⚠️ | ✅ **明确沙盒策略+禁止命令列表** |
| 规划模式 | 隐式 | ❌ | ❌ | ✅ **显式规划模式，用户可介入确认** |

### 差异化定位

1. **系统化 Harness 架构**：不是单点功能增强，而是将六大组件作为系统工程构建，架构可扩展性强。
2. **显式规划-执行闭环**：长链路任务先规划、后执行，用户有充分的控制和知情权。
3. **指针索引 Context Manager**：区别于竞品的隐式上下文管理，提供可观测、可干预的上下文管理机制。
4. **Verification Hooks 保障**：独立的执行结果验证层，降低 AI 幻觉风险。

## 10. Acceptance Criteria

### 功能验收

- [ ] 应用启动后显示对话界面，可输入自然语言指令
- [ ] 短任务（≤3步）直接执行，返回正确结果
- [ ] 长链路任务进入规划模式，展示执行方案，用户确认后执行
- [ ] `file_read`、`file_write`、`bash_execute` 核心工具可正常工作
- [ ] 中/高风险工具有对应的权限确认弹窗
- [ ] 上下文超过阈值时自动压缩，不丢失关键指针
- [ ] 工具执行结果不符合预期时，系统给出警告提示
- [ ] 应用关闭后重启，上下文可恢复（上一次会话 Memory 持久化）

### 技术验收

- [ ] Electron 应用在 Windows 环境下稳定运行，无崩溃
- [ ] LLM API 调用异常时有合理的错误提示，不闪退
- [ ] 沙盒内执行的 Bash 命令不超出授权范围
- [ ] UI 响应时间 < 500ms（不含 LLM API 调用延迟）

### 安全验收

- [ ] `rm -rf /` 类危险命令在沙盒中被阻止
- [ ] 文件操作限制在工作目录内，超出范围报错
- [ ] API Key 以加密形式存储在本地

## 11. Risks and Open Questions

### 已知风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| LLM API 延迟影响体验 | 中 | 提供流式响应展示，逐步加载结果 |
| 沙盒安全性难以100%保障 | 高 | 配置禁止命令白名单，定期审计 |
| 上下文压缩算法可能导致信息丢失 | 中 | MVP 使用简单压缩策略，v2 引入语义压缩 |
| Electron 在 Windows 上的性能 | 低 | 关注主进程与渲染进程分离，避免阻塞 |

### Open Questions（待确认）

| 问题 | 优先级 | 说明 |
|---|---|---|
| MVP 接入的 LLM 模型具体是哪家？（OpenAI / Anthropic / 自部署） | P0 | 建议先支持 OpenAI API 兼容接口 |
| 用户工作目录的根目录由谁指定？用户配置 or 首次启动时选择？ | P1 | 建议支持启动时选择 + 设置中可改 |
| API Key 的获取方式？用户自行填写还是提供授权流程？ | P1 | MVP 建议用户自行填写，设置界面入口 |
| Verification Hooks 的警告展示形式？内联在对话中还是单独面板？ | P2 | MVP 建议内联，保持对话连贯性 |
| 是否需要支持流式输出（Streaming）？ | P1 | 建议支持，提升长回复体验 |

## 12. Handoff

### 交付物

1. 本 PRD 文档（`2026-04-17-harness-engineering-prd.md`）
2. 后续由 `dev` 基于本文档产出技术设计和实现计划

### 对 `dev` 的实现提示

1. **优先实现 Tool Call Loop**：这是 MVP 的核心，建议先完成工具执行框架和规划-执行双模式，再逐步完善其他组件。
2. **沙盒安全先行**：在实现 `bash_execute` 之前，先完成沙盒管理机制，明确禁止命令列表。
3. **System Prompt 引擎可独立测试**：可以在无 LLM 的情况下用 Mock 数据测试 Prompt 拼接逻辑。
4. **Context Manager 的指针设计**：建议指针使用自增 ID，内容存储使用 Content-addressable 方式，便于去重和关联。
5. **Electron IPC 命名规范**：建议使用 `module:action` 格式，如 `tool:execute`、`context:compress`。

### 依赖

- Node.js >= 18
- Electron >= 28
